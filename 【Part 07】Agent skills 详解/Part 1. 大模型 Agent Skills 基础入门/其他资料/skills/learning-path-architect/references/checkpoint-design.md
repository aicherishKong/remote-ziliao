# Checkpoint Design (检查点设计方法)

## 目标
为每个学习阶段设计验证标准，确保学习效果。

## 执行步骤

### 1. 为每个阶段设计 3-5 个检查点
- 检查点应该覆盖该阶段的核心能力
- 检查点应该具有可验证性
- 检查点应该由易到难排列

### 2. 检查点类型

#### 理论检查
- 能否解释核心概念？
- 能否说出关键特性和区别？
- 能否理解技术的适用场景？

**示例**:
- ✓ 能否用一句话解释 [核心概念]?
- ✓ 能否说出与 [类似技术] 的 3 个区别?
- ✓ 能否解释 [技术特性] 的工作原理?

#### 实践检查
- 能否独立完成某个任务？
- 能否运行和验证示例代码？
- 能否解决实际问题？

**示例**:
- ✓ 能否独立运行 Hello World 示例?
- ✓ 能否独立完成 [场景1] 的案例实现?
- ✓ 能否独立完成完整项目?

#### 应用检查
- 能否解决实际问题？
- 能否优化性能？
- 能否处理常见错误？

**示例**:
- ✓ 能否优化 [性能问题]?
- ✓ 能否处理常见的线上问题?
- ✓ 能否部署到生产环境?

### 3. 提供自测题目和参考答案
- 每个检查点应该有对应的自测题目
- 题目应该具有明确的评判标准
- 提供参考答案或评分标准

## 输出格式

```
学习检查点:

阶段1 检查点:
  ✓ 能否用一句话解释 [核心概念]?
  ✓ 能否独立运行 Hello World 示例?
  ✓ 能否说出与 [类似技术] 的 3 个区别?

阶段2 检查点:
  ✓ 能否独立完成 [场景1] 的案例实现?
  ✓ 能否解释 [进阶概念] 的工作原理?
  ✓ 能否优化 [性能问题]?

阶段3 检查点:
  ✓ 能否独立完成完整项目?
  ✓ 能否部署到生产环境?
  ✓ 能否处理常见的线上问题?
```

## 设计原则

- **可验证性**: 每个检查点都应该有明确的验证方法
- **覆盖性**: 检查点应该覆盖该阶段的核心能力
- **递进性**: 检查点应该由易到难排列
- **实用性**: 检查点应该与实际应用场景相关
- **明确性**: 检查点的描述应该清晰明确，避免模糊表述

## 常见检查点模板

### 基础认知阶段
- 能否用一句话解释 [核心概念]?
- 能否说出 [技术] 的 3-5 个核心特性?
- 能否说出与 [类似技术] 的主要区别?
- 能否独立运行最小示例?
- 能否解释 [核心术语] 的含义?

### 进阶应用阶段
- 能否独立完成 [典型场景] 的实现?
- 能否解释 [进阶概念] 的工作原理?
- 能否识别和解决 [常见错误]?
- 能否优化 [性能问题]?
- 能否在生产环境中应用?

### 项目实战阶段
- 能否独立完成完整项目?
- 能否拆解项目为独立模块?
- 能否设计测试用例?
- 能否部署到生产环境?
- 能否处理线上问题?
